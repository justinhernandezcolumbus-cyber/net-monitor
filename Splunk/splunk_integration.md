# Splunk Integration Workflow

## Overview

Home internet performance is difficult to evaluate without objective data. Anecdotal impressions of slow speeds or dropped connections are hard to quantify, and it's even more difficult to compare performance across different connection types within a household.

I needed to measure the actual performance of my network with real metrics to understand the quality of my service and determine whether to continue with my current provider. Three questions drove this analysis:

- **Should I get a new provider?**
- **Is the service matching what my provider advertises?**
- **What times of day is performance worst?**

This workflow connects three components: a Python monitoring script, a structured CSV log, and Splunk Enterprise for search, visualization, and alerting.

---

## Step 1 — Configure Data Inputs

The CSV files generated by `monitor.py` are ingested into Splunk as file-based data inputs. Each row in the CSV becomes a searchable event in Splunk, with the column headers automatically recognized as field names.

**Steps (Splunk Enterprise 10.x):**

1. Navigate to **Settings → Data Inputs → Files & Directories**
2. Click **New Local File & Directory**
3. Enter the fully qualified path to your CSV file
4. Select **Index Once**

> Since these files represent a fixed monitoring period rather than a live data stream, Index Once is the appropriate choice. It ingests the snapshot dataset as-is, suitable for retrospective analysis of a defined time window. If the tool were running continuously and writing new rows in real time, Continuously Monitor would be the correct option instead.

5. Click **Next**
6. On the Set Source Type screen, click **Save As** and name it `net_monitor_desktop` or `net_monitor_macbook` depending on which file you're adding

> Using a descriptive source type name rather than the default `csv` allows you to filter by device in SPL queries, making wired vs. wireless comparisons straightforward.

7. Click **Next**
8. Leave Input Settings at defaults
9. Review the summary and click **Submit**

Repeat for each CSV file. Both source types will be searchable from the same index.

## Step 2 — Verify the Data

Before building visualizations, confirm that Splunk is reading the fields correctly.

```
sourcetype=net_monitor_desktop OR sourcetype=net_monitor_macbook
| table timestamp check_type avg_latency_ms jitter packet_loss_pct download_mbps upload_mbps alert_triggered alert_reason
```

Each column from the CSV should appear as a separate field. Ping rows will have empty `download_mbps` and `upload_mbps` values — that is expected behavior, not a data issue.

## Step 3 — SPL Queries

All queries use a shared base search that covers both devices and a defined time window. Adjust the `earliest` and `latest` values to match your monitoring period.

**Base search (reused across all panels):**

```
sourcetype=net_monitor_desktop OR sourcetype=net_monitor_macbook earliest="04/06/2026:00:00:00" latest="04/06/2026:23:59:59"
```

---

### Panel 1 — Average Latency by Device Over Time

Compares average round-trip latency between the wired desktop and wireless MacBook over time. Uses 30-minute buckets to smooth the data.

```
sourcetype=net_monitor_desktop OR sourcetype=net_monitor_macbook earliest="04/06/2026:00:00:00" latest="04/06/2026:23:59:59" check_type=ping
| timechart span=30m avg(avg_latency_ms) as "Avg Latency (ms)" by sourcetype
```

**Visualization:** Line chart, split by sourcetype

---

### Panel 2 — Average Jitter by Device Over Time

Jitter measures the difference between minimum and maximum latency per check — a more revealing stability metric than average latency alone. A connection can look acceptable on average while experiencing severe instability.

```
sourcetype=net_monitor_desktop OR sourcetype=net_monitor_macbook earliest="04/06/2026:00:00:00" latest="04/06/2026:23:59:59" check_type=ping
| timechart span=30m avg(jitter) as "Avg Jitter (ms)" by sourcetype
```

**Visualization:** Line chart, split by sourcetype

---

### Panel 3 — Average Download Speed by Device Over Time

Compares download throughput between devices over time. Speed tests run every 30 minutes so hourly buckets are used to avoid sparse data points.

```
sourcetype=net_monitor_desktop OR sourcetype=net_monitor_macbook earliest="04/06/2026:00:00:00" latest="04/06/2026:23:59:59" check_type=speed
| timechart span=1h avg(download_mbps) as "Avg Download (Mbps)" by sourcetype
```

**Visualization:** Line chart, split by sourcetype

---

### Panel 4 — Net Monitor Alerts Summary

Shows how many threshold breaches occurred per device across the monitoring period, alongside total checks run. Provides a reliability summary at a glance.

```
sourcetype=net_monitor_desktop OR sourcetype=net_monitor_macbook earliest="04/06/2026:00:00:00" latest="04/06/2026:23:59:59"
| stats count(eval(alert_triggered="True")) as "Alerts Fired" count as "Total Checks" by sourcetype
```

**Visualization:** Table

## Step 4 — Dashboard

The four panels above were combined into a single Splunk dashboard titled **Net Monitor Dashboard**. Screenshots are available in this directory.

To save a search as a dashboard panel:
1. Run the query in Search & Reporting
2. Click **Save As → Dashboard Panel**
3. Select an existing dashboard or create a new one
4. Name the panel and save

## Key Findings

| Metric | Desktop (Wired) | MacBook (Wireless) |
|---|---|---|
| Avg Latency | ~25ms | ~28ms |
| Max Latency | 130ms | 68ms |
| Avg Jitter | ~28ms | ~29ms |
| Max Jitter | 420ms | 236ms |
| Avg Download | 87.2 Mbps | 88.8 Mbps |
| Alerts Fired | 1 | 0 |

**Notable observations:**

- Jitter spikes occurred simultaneously on both devices — wired and wireless — at the same times of day. This points to instability in the provider's network upstream rather than in home hardware or the wireless connection.
- The wired desktop showed a higher maximum latency spike (130ms) and higher peak jitter (420ms) than the wireless MacBook, which is counterintuitive and further supports the conclusion that the instability is not connection-type dependent.
- Download speeds on both devices remained above the 50 Mbps threshold throughout the monitoring period, suggesting the provider is delivering advertised throughput even when latency is degraded.

**Conclusion:** The data does not support switching providers on the basis of speed — throughput is adequate. However, the jitter profile suggests network instability at the provider level that may warrant monitoring over a longer period before making a final decision.
